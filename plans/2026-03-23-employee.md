## Wave Execution Order

### Backend Waves

**Wave 1 — Domain Entities + Events + Errors** (no dependencies)
- Position entity + 3 events
- Employee entity + CPF value object + 5 events
- All domain error classes (11 total)
- Can be parallelized: positions subdomain ∥ employees subdomain

**Wave 2 — Use Cases + Repository Interfaces** (depends on Wave 1)
- Positions repository interface + 5 use cases (create, edit, delete, list, list-audit-logs)
- Employees repository interface + 8 use cases (create, edit, inactivate, activate, delete, list, find-by-id, list-audit-logs)
- QueryBus query contract + handler (FindEmployee)
- Can be parallelized: positions use cases ∥ employees use cases

**Wave 3 — Test Infrastructure + Unit Tests** (depends on Wave 2)
- Factories (make-position, make-employee)
- In-memory repositories (positions, employees)
- Unit tests for all 13 use cases
- Can be parallelized: position tests ∥ employee tests

**Wave 4 — Prisma + Infra DB** (depends on Wave 2)
- Prisma schema changes (Position + Employee models + EmployeeStatus enum)
- Migration
- Seed data (predefined positions)
- Mappers (prisma-position, prisma-employee)
- Prisma repositories
- Database module

**Wave 5 — HTTP Layer** (depends on Wave 4)
- DTOs (request, response, error)
- Presenters
- Error filter
- Controllers (13 total)
- HTTP module
- Events module
- QueryBus module update
- AppModule registration
- E2E tests for all controllers

### Frontend Waves

**Wave 6 — Schemas + API** (no backend dependency at runtime, follows contract)
- Position schemas + API functions
- Employee schemas + API functions
- Can be parallelized: positions ∥ employees

**Wave 7 — Store + Actions** (depends on Wave 6)
- Position store + server actions
- Employee store + server actions

**Wave 8 — Components** (depends on Wave 7)
- Position components (table, forms, sheets, dialogs, ui, audit maps)
- Employee components (table, forms, sheets, dialogs, ui, audit maps)
- Can be parallelized: positions ∥ employees

**Wave 9 — Pages + Navigation** (depends on Wave 8)
- Employee pages (list, detail, audit)
- Position pages (list, audit)
- Loading skeletons
- Sidebar navigation update

**Wave 10 — Playwright E2E** (depends on Wave 9)
- MSW handlers
- E2E test specs

## Integration Points

1. **QueryBus:** Register `FindEmployeeHandler` in `QueryBusModule` — for future FieldTicket reference
2. **Audit:** Domain events auto-consumed by existing audit subscriber infrastructure
3. **CASL:** Add Employee and Position to ability factory (owner/manager/others permissions)
4. **Sidebar:** Add "Funcionários" and "Cargos" navigation items
5. **Seed:** Add predefined positions (Owner, Gerente, Operador, Tratorista, Auxiliar)

## Notes

- FieldTicket `operatorName` → `employeeId` migration is OUT OF SCOPE for this implementation. Will be a separate small-change after Employee domain is live.
- User reference is optional — validate userId via QueryBus if provided.
- Delete employee checks for cross-domain references (FieldTicket) — for now, since FieldTicket doesn't reference Employee yet, hard delete is always possible. The check infrastructure should be in place for when the migration happens.
