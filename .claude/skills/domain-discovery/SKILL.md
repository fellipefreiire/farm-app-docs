# Domain Discovery

Facilitates a structured conversation to define everything needed to implement a domain. The output is a `docs/rules/<domain>.md` file that becomes the authoritative reference for that domain.

This skill must run **before** `/new-domain` or `/feature` if the domain rules file is missing.

---

## Process

Ask questions **one at a time**. Wait for the answer before asking the next. Use the answers to build the rules file incrementally.

### 1. Domain Overview

- What is the name of this domain? (used as the code name — English, singular, PascalCase)
- What is its responsibility in one sentence?
- What does it NOT handle? (explicit boundaries)

### 2. Entities

For the primary entity:
- What are its attributes? (name, type, required/optional)
- What are the business invariants? (rules that must always be true: e.g., "price must be positive")
- Does it have an identity? (UUID generated by the system, or provided externally?)
- Does it have status/state transitions? If yes, describe the lifecycle

For each related entity:
- Same questions as above
- What is the relationship? (one-to-many, many-to-many, embedded)

### 3. Value Objects

- Are there attributes that represent concepts with their own validation? (e.g., Email, CPF, Money, Coordinate)
- List them with their validation rules

### 4. Use Cases

For each operation the domain supports:
- What triggers it? (user action, event from another domain, scheduled job)
- What are the inputs?
- What are the business rules that must be enforced?
- What are the possible outcomes? (success result + all error cases)
- Does it emit domain events? If yes, which ones?

### 5. Cross-Domain Relationships

- Which existing domains does this one depend on?
- Which domains will consume events from this one?
- Are there shared entities (referenced by ID only — never imported directly)?

### 6. Authorization

- Who can perform each use case? (roles, ownership rules)
- Are there resource-level restrictions? (e.g., a user can only see their own records)

### 7. Edge Cases and Constraints

- What are the known edge cases?
- Are there uniqueness constraints? (e.g., email must be unique per tenant)
- Are there soft-delete requirements?
- Are there audit trail requirements?

---

## Output

After collecting all answers, write `docs/rules/<domain>.md`:

```markdown
# <Domain> — Business Rules

## Responsibility
<one sentence>

## Out of scope
<what this domain does NOT handle>

## Entities

### <Entity>
| Attribute | Type | Required | Rules |
|-----------|------|----------|-------|
| id | UUID | yes | generated by system |
| ... | | | |

**Invariants:**
- <rule>

**Lifecycle:** <states and transitions, or "no state machine">

## Value Objects
- **<ValueObject>** — <what it represents, validation rules>

## Use Cases

### <ActionEntity> (e.g., CreateOrder)
- **Trigger:** <what initiates this>
- **Input:** <fields>
- **Rules:** <business rules enforced>
- **Success:** <what is returned>
- **Errors:** <typed error cases>
- **Events emitted:** <DomainEvent names, or "none">

## Cross-Domain Dependencies
- Depends on: <DomainA> (by ID reference only)
- Emits events consumed by: <DomainB>

## Authorization
- <UseCase>: <who can execute, ownership rules>

## Constraints
- <uniqueness, soft-delete, audit trail requirements>

## Edge Cases
- <known edge cases>

## Open Questions
- <anything still undefined — to be resolved before implementation>
```

---

## Completion

Present the filled rules file to the user and ask:
1. Is everything correct?
2. Are there open questions that need answering before implementation?

If confirmed → ask the user: "Domain rules are ready. Do you want me to run `/new-domain` to implement this domain, `/feature` for a specific feature, or save these rules for later?" If the user wants to save for later, acknowledge completion: "Rules saved. Mention the domain name when you're ready to implement."

Then list the coding patterns relevant for implementing this domain:
- `docs/coding-patterns/backend/entity.md` — for entities and value objects defined above
- `docs/coding-patterns/backend/use-case.md` — for each use case listed
- `docs/coding-patterns/backend/repository.md` — for the repository interface and implementations
- `docs/coding-patterns/backend/controller.md` — for the HTTP layer
- `docs/coding-patterns/backend/events.md` — if domain events were defined
- `docs/coding-patterns/frontend/` — for the frontend slice
- `docs/coding-patterns/frontend/design-system.md` — for color tokens, typography scale, and dark mode

If corrections needed → adjust and confirm again.
