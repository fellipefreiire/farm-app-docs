# Schedule — Business Rules

## Responsibility

Lightweight grouping entity that represents the operation plan for a harvest on a field. Status is **automatic** — driven by FieldTicket lifecycle, not by manual user actions. The only explicit user action on status is **cancellation**. Enforces the constraint: **max 1 ACTIVE schedule per field at a time**.

## Out of scope

- Operation details, inputs, dosages (owned by FieldTicket domain — Schedule queries via QueryBus)
- Execution control and tracking (owned by FieldTicket domain)
- Inventory stock deduction (owned by Inventory domain, triggered by FieldTicket finalization)
- Field and crop management (owned by Field and Crop domains)
- Financial tracking (owned by Financial domain, post-MVP)

## Entities

### Schedule

| Attribute | Type | Required | Rules |
|-----------|------|----------|-------|
| id | UUID | yes | generated by system |
| harvestId | UUID | yes | references Harvest (1:1 — one schedule per harvest) |
| status | enum | yes | PLANNED, ACTIVE, UNDER_REVIEW, COMPLETED, CANCELLED — see lifecycle below |
| reviewNotes | string | no | filled during UNDER_REVIEW → COMPLETED transition |
| reviewedAt | DateTime | no | set when review is confirmed |
| createdAt | DateTime | yes | set on creation |
| updatedAt | DateTime | yes | set on every update |
| deletedAt | DateTime | no | set on soft delete |

**Invariants:**
- One Schedule per Harvest (1:1 relationship)
- Harvest must exist and have status UNSCHEDULED to create a Schedule
- **Max 1 ACTIVE schedule per field at a time** — if a field already has an ACTIVE schedule, new schedules are created as PLANNED
- Schedule period is inherited from Harvest (startDate, expectedEndDate) — no separate dates
- Status transitions are automatic, except cancellation (explicit user action)

**Lifecycle:**

```
PLANNED ──(user selects during review/cancel flow)──→ ACTIVE
ACTIVE  ──(all FieldTickets terminal)──→ UNDER_REVIEW
UNDER_REVIEW ──(user reviews and confirms)──→ COMPLETED
UNDER_REVIEW ──(user adds new FieldTicket)──→ ACTIVE
PLANNED / ACTIVE ──(user cancels)──→ CANCELLED
```

- **PLANNED:** schedule created for a field that already has an ACTIVE schedule. Waiting in queue. Users CAN add FieldTickets to plan operations in advance.
- **ACTIVE:** the schedule being executed. Only 1 per field. Harvest is ACTIVE.
- **UNDER_REVIEW:** all FieldTickets reached terminal state (COMPLETED/CANCELLED). User reviews: checks if operations are complete, leaves notes, then confirms or adds missing operations (reverts to ACTIVE).
- **COMPLETED:** review confirmed. Harvest → COMPLETED. User chose next PLANNED schedule (if any) to activate.
- **CANCELLED:** user explicitly cancelled. DRAFT/REVIEWED tickets auto-cancelled. COMPLETED tickets preserved. Harvest → CANCELLED.

**Status determination on creation:**
1. Find the field via `Harvest.fieldId`
2. Check if any ACTIVE schedule exists for that field (across all harvests)
3. If no ACTIVE schedule → born **ACTIVE**, linked Harvest → ACTIVE
4. If ACTIVE schedule exists → born **PLANNED**

**Operations:**
Schedule does NOT own operations directly. Operations are FieldTickets linked by `scheduleId`. To list operations for a schedule, use QueryBus (`FindFieldTicketsByScheduleIdQuery`).

## Use Cases

### CreateSchedule
- **Trigger:** user action
- **Input:** harvestId
- **Rules:** Harvest must exist with status UNSCHEDULED. One Schedule per Harvest (1:1). Check field for existing ACTIVE schedule to determine initial status.
- **Success:** Schedule created with status ACTIVE (if field has no active schedule) or PLANNED (if field already has active schedule). If ACTIVE, linked Harvest → ACTIVE.
- **Errors:** HarvestNotFoundError, HarvestNotUnscheduledError, ScheduleAlreadyExistsForHarvestError
- **Events emitted:** ScheduleCreatedEvent. If born ACTIVE: ScheduleActivatedEvent (→ Harvest ACTIVE).

### EditSchedule
- **Trigger:** user action
- **Input:** scheduleId, harvestId (optional — allows re-linking to a different UNSCHEDULED harvest)
- **Rules:** Schedule must exist. If changing harvestId, new Harvest must be UNSCHEDULED. Cannot edit if COMPLETED, CANCELLED, or UNDER_REVIEW.
- **Success:** Schedule updated
- **Errors:** ScheduleNotFoundError, HarvestNotFoundError, HarvestNotUnscheduledError, ScheduleNotEditableError
- **Events emitted:** ScheduleUpdatedEvent

### DeleteSchedule
- **Trigger:** user action
- **Input:** scheduleId
- **Rules:** If no FieldTickets exist, or FieldTickets exist but none are printed/completed → hard delete (FieldTickets in DRAFT/REVIEWED are cascade deleted). If FieldTickets are printed or completed → soft delete. When hard deleted, Harvest reverts to UNSCHEDULED.
- **Success:** Schedule deleted (hard or soft)
- **Errors:** ScheduleNotFoundError
- **Events emitted:** ScheduleDeletedEvent

### ReviewSchedule
- **Trigger:** user action (replaces CompleteSchedule)
- **Input:** scheduleId, reviewNotes (optional), nextScheduleId (optional — which PLANNED schedule to activate next)
- **Rules:** Status must be UNDER_REVIEW. Sets reviewNotes and reviewedAt. Transitions to COMPLETED. Linked Harvest → COMPLETED. If nextScheduleId provided, that schedule → ACTIVE and its Harvest → ACTIVE.
- **Success:** Schedule → COMPLETED, next schedule promoted if selected
- **Errors:** ScheduleNotFoundError, ScheduleNotUnderReviewError, NextScheduleNotFoundError, NextScheduleNotPlannedError
- **Events emitted:** ScheduleReviewedEvent, ScheduleCompletedEvent (→ Harvest COMPLETED). If next promoted: ScheduleActivatedEvent (→ next Harvest ACTIVE).

### CancelSchedule
- **Trigger:** user action (only explicit status action)
- **Input:** scheduleId, reason (mandatory), nextScheduleId (optional)
- **Rules:** Status must be PLANNED or ACTIVE. **PRINTED tickets must be resolved first** — use case returns error if any PRINTED tickets exist, listing their IDs for the frontend to show the resolution flow. After PRINTED resolved: DRAFT/REVIEWED tickets are auto-cancelled. COMPLETED tickets are preserved. Harvest → CANCELLED.
- **Pre-condition:** No FieldTickets with status PRINTED (must be finalized or cancelled individually first)
- **Success:** Schedule → CANCELLED, non-terminal tickets → CANCELLED, Harvest → CANCELLED. If nextScheduleId provided, that schedule → ACTIVE.
- **Errors:** ScheduleNotFoundError, ScheduleNotCancellableError, UnresolvedPrintedTicketsError (includes ticket IDs)
- **Events emitted:** ScheduleCancelledEvent (→ Harvest CANCELLED). FieldTicketCancelledEvent per auto-cancelled ticket. If next promoted: ScheduleActivatedEvent.

### GetSchedule
- **Trigger:** user action
- **Input:** scheduleId
- **Rules:** only return non-soft-deleted records. Operations (FieldTickets) fetched via QueryBus.
- **Success:** Schedule with Harvest info (field, crop type, variety, dates) + list of FieldTickets for this schedule
- **Errors:** ScheduleNotFoundError
- **Events emitted:** none

### ListSchedules
- **Trigger:** user navigates to schedules page
- **Input:** pagination params, optional filters (status, fieldId, harvestId)
- **Rules:** only return non-soft-deleted records, paginated
- **Success:** paginated list of schedules with Harvest summary (field name, crop type, variety, dates, status)
- **Errors:** none
- **Events emitted:** none

---

### CopyScheduleOperationsToDay
- **Trigger:** user action (copy operations from one day to another)
- **Input:** scheduleId, sourceDate, targetDate, mode (add/replace)
- **Rules:** Both dates must be within Harvest period. Schedule must be PLANNED or ACTIVE (not UNDER_REVIEW, COMPLETED, or CANCELLED). Queries FieldTickets for sourceDate via QueryBus, creates new DRAFT FieldTickets on targetDate.
- **Success:** New DRAFT FieldTickets created on targetDate
- **Errors:** ScheduleNotFoundError, ScheduleNotEditableError, OperationDateOutOfRangeError, NoOperationsOnSourceDateError
- **Events emitted:** FieldTicketCreatedEvent (one per copied ticket, emitted by FieldTicket domain)

### CopySchedule
- **Trigger:** user action (copy entire schedule to create a new one for a different harvest)
- **Input:** sourceScheduleId, targetHarvestId
- **Rules:** Source Schedule must exist (any status — including CANCELLED). Target Harvest must exist with status UNSCHEDULED. Copies ALL operations as DRAFT regardless of their original status.
- **Success:** New Schedule created + DRAFT FieldTickets created with copied data
- **Errors:** ScheduleNotFoundError, HarvestNotFoundError, HarvestNotUnscheduledError, ScheduleAlreadyExistsForHarvestError
- **Events emitted:** ScheduleCreatedEvent, FieldTicketCreatedEvent (per copied ticket)

## Event Subscribers (auto-transitions)

### OnFieldTicketTerminalCheckSchedule
- **Listens to:** FieldTicketCompletedEvent, FieldTicketCancelledEvent
- **Logic:** Find the schedule for this FieldTicket. If schedule is ACTIVE, check if ALL FieldTickets for this schedule are in terminal state (COMPLETED or CANCELLED). If yes → transition schedule to UNDER_REVIEW.
- **Events emitted:** ScheduleUnderReviewEvent

### OnFieldTicketCreatedRevertReview
- **Listens to:** FieldTicketCreatedEvent
- **Logic:** If the schedule is UNDER_REVIEW and a new FieldTicket is created for it → revert schedule to ACTIVE.
- **Events emitted:** ScheduleActivatedEvent

## Cross-Domain Dependencies

- **Depends on:** Harvest (by `harvestId` — 1:1, inherits field/crop/dates), Field (indirectly via Harvest, to check active schedule per field), FieldTicket (via QueryBus — `FindFieldTicketsByScheduleIdQuery` to list operations)
- **Emits events consumed by:** Crop (ScheduleActivatedEvent → activate Harvest, ScheduleCompletedEvent → complete Harvest, ScheduleCancelledEvent → cancel Harvest), FieldTicket (ScheduleCancelledEvent triggers auto-cancel of DRAFT/REVIEWED tickets), Audit (all mutation events)

## Constraints

- One Schedule per Harvest (1:1) — enforced by unique constraint on harvestId
- Max 1 ACTIVE schedule per field — enforced by CreateSchedule logic (check across harvests for the same field)
- Schedule does NOT store operations — FieldTickets are the operations, linked by scheduleId
- All mutations audited via domain events
- Hard delete cascades to draft/reviewed FieldTickets (via event to FieldTicket domain)
- Soft delete when printed/completed FieldTickets exist
- Schedule status drives Harvest status (ACTIVE → ACTIVE, COMPLETED → COMPLETED, CANCELLED → CANCELLED)

## Edge Cases

- User creates Schedule for a field that already has an ACTIVE schedule → new schedule is PLANNED
- User creates Schedule for a field with no ACTIVE schedule → new schedule is ACTIVE, Harvest activated
- Schedule with 0 FieldTickets never auto-transitions to UNDER_REVIEW (no trigger) → user must cancel manually if unwanted
- User adds a FieldTicket to an UNDER_REVIEW schedule → schedule reverts to ACTIVE
- User cancels ACTIVE schedule with PRINTED tickets → error: must resolve (finalize or cancel) each PRINTED ticket first
- User cancels ACTIVE schedule with only DRAFT/REVIEWED tickets → auto-cancelled, schedule → CANCELLED
- User cancels PLANNED schedule → straightforward, no ticket resolution needed (PLANNED tickets auto-cancelled)
- User reviews schedule and chooses next PLANNED → current COMPLETED, next ACTIVE, harvests updated accordingly
- User reviews schedule with no PLANNED available → COMPLETED, no promotion, option to create new
- User copies a CANCELLED schedule → all operations (including originally COMPLETED ones) copied as DRAFT
- Concurrent Schedule creation for same Harvest → database unique constraint prevents duplicates

## Harvest Domain Changes Required

1. **Harvest lifecycle driven by Schedule:**
   ```
   UNSCHEDULED → ACTIVE (via ScheduleActivatedEvent)
   ACTIVE → COMPLETED (via ScheduleCompletedEvent)
   ACTIVE → CANCELLED (via ScheduleCancelledEvent)
   UNSCHEDULED → CANCELLED (manual, if harvest is abandoned before scheduling)
   ```
2. **Remove manual `ActivateHarvest` use case** — activation is driven by Schedule domain
3. **Add/update event subscribers** for ScheduleActivatedEvent, ScheduleCompletedEvent, ScheduleCancelledEvent
4. **`CompleteHarvest` driven by Schedule** — Harvest completes when Schedule completes (after review), not manually
5. **Initial status on creation** — Harvest is created with status `UNSCHEDULED` (no schedule yet)
