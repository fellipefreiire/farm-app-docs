# Schedule — Business Rules

## Responsibility

Aggregate and centralize the operation planning for a harvest on a field — which operations (spraying, fertigation), which inputs, on which days. The Schedule is the source of truth for what happens in the physical space.

## Out of scope

- Execution control and tracking (owned by FieldTicket domain)
- Inventory stock deduction (owned by Inventory domain, triggered by FieldTicket finalization)
- Field and crop management (owned by Field and Crop domains)
- Financial tracking (owned by Financial domain, post-MVP)

---

## Entities

### Schedule

| Attribute | Type | Required | Rules |
|-----------|------|----------|-------|
| id | UUID | yes | generated by system |
| harvestId | UUID | yes | references Harvest (1:1 — one schedule per harvest) |
| status | enum | yes | PLANNED, ACTIVE, COMPLETED, CANCELLED — defaults to PLANNED |
| createdAt | DateTime | yes | set on creation |
| updatedAt | DateTime | yes | set on every update |
| deletedAt | DateTime | no | set on soft delete |

**Invariants:**
- One Schedule per Harvest (1:1 relationship)
- Harvest must exist and have status UNSCHEDULED to create a Schedule
- When Schedule status changes to ACTIVE, the linked Harvest status changes to ACTIVE (via domain event)
- When Schedule is cancelled, the linked Harvest reverts to UNSCHEDULED (via domain event)
- When Schedule is completed, the linked Harvest can be completed manually by the user
- Schedule period is inherited from Harvest (startDate, expectedEndDate) — no separate dates

**Lifecycle:**

```
PLANNED → ACTIVE → COMPLETED
PLANNED → CANCELLED
ACTIVE → CANCELLED
```

- PLANNED: schedule created, operations being defined, not yet in execution
- ACTIVE: schedule in execution (operations happening in the field)
- COMPLETED: all operations finished (manual — user confirms completion)
- CANCELLED: schedule abandoned — Harvest reverts to UNSCHEDULED, allowing a new Schedule

---

### ScheduleOperation

| Attribute | Type | Required | Rules |
|-----------|------|----------|-------|
| id | UUID | yes | generated by system |
| scheduleId | UUID | yes | references Schedule |
| date | Date | yes | must be within the Harvest period (startDate ≤ date ≤ expectedEndDate) |
| type | enum | yes | SPRAYING, FERTIGATION |
| createdAt | DateTime | yes | set on creation |
| updatedAt | DateTime | yes | set on every update |

**Invariants:**
- Date must fall within the Harvest period (startDate to expectedEndDate)
- A day can have multiple operations (same or different types)
- Must have at least one ScheduleOperationInput

**Lifecycle:** no state machine

---

### ScheduleOperationInput

| Attribute | Type | Required | Rules |
|-----------|------|----------|-------|
| id | UUID | yes | generated by system |
| operationId | UUID | yes | references ScheduleOperation |
| inputId | UUID | yes | references Input (Inventory domain, by ID) |
| dosagePer100L | decimal | yes | must be > 0, dosage per 100L of water |
| createdAt | DateTime | yes | set on creation |
| updatedAt | DateTime | yes | set on every update |

**Invariants:**
- dosagePer100L must be > 0
- inputId must reference an existing, active Input
- Dosage unit is always the "smaller" unit relative to the Input's unitOfMeasure: kg→g, L→mL, g→g, mL→mL, un→un
- An operation can have multiple inputs

**Lifecycle:** no state machine

---

## Value Objects

- **OperationType** — enum: SPRAYING, FERTIGATION. Fixed in the system for MVP, not user-configurable.
- **DosagePer100L** — decimal, must be > 0. Represents quantity of input per 100L of water. Display unit is always the smaller unit (kg→g, L→mL).

## Use Cases

### CreateSchedule
- **Trigger:** user action
- **Input:** harvestId
- **Rules:** Harvest must exist with status UNSCHEDULED. One Schedule per Harvest (1:1).
- **Success:** Schedule created with status PLANNED. Harvest status changes to PLANNED (via domain event — not yet, MVP keeps Harvest status change manual? No — see lifecycle: Harvest goes from UNSCHEDULED to having a schedule, but stays as-is until Schedule is activated).
- **Errors:** HarvestNotFoundError, HarvestNotUnscheduledError, ScheduleAlreadyExistsForHarvestError
- **Events emitted:** ScheduleCreatedEvent

### EditSchedule
- **Trigger:** user action
- **Input:** scheduleId, harvestId (optional — allows re-linking to a different UNSCHEDULED harvest)
- **Rules:** Schedule must exist. If changing harvestId, new Harvest must be UNSCHEDULED. Cannot edit if COMPLETED or CANCELLED.
- **Success:** Schedule updated
- **Errors:** ScheduleNotFoundError, HarvestNotFoundError, HarvestNotUnscheduledError, ScheduleNotEditableError
- **Events emitted:** ScheduleUpdatedEvent

### DeleteSchedule
- **Trigger:** user action
- **Input:** scheduleId
- **Rules:** If no FieldTickets exist, or FieldTickets exist but none are printed/finalized → hard delete (cascade: all operations, operation inputs, and draft FieldTickets). If FieldTickets are printed or finalized → soft delete. When hard deleted, Harvest reverts to UNSCHEDULED.
- **Success:** Schedule deleted (hard or soft)
- **Errors:** ScheduleNotFoundError
- **Events emitted:** ScheduleDeletedEvent

### ActivateSchedule
- **Trigger:** user action
- **Input:** scheduleId
- **Rules:** status must be PLANNED. Linked Harvest status changes to ACTIVE (via domain event).
- **Success:** Schedule status → ACTIVE, Harvest status → ACTIVE
- **Errors:** ScheduleNotFoundError, ScheduleNotPlannedError
- **Events emitted:** ScheduleActivatedEvent (consumed by Crop domain to activate Harvest)

### CompleteSchedule
- **Trigger:** user action
- **Input:** scheduleId
- **Rules:** status must be ACTIVE.
- **Success:** Schedule status → COMPLETED
- **Errors:** ScheduleNotFoundError, ScheduleNotActiveError
- **Events emitted:** ScheduleCompletedEvent

### CancelSchedule
- **Trigger:** user action
- **Input:** scheduleId
- **Rules:** status must be PLANNED or ACTIVE. Harvest reverts to UNSCHEDULED (via domain event), allowing a new Schedule to be created.
- **Success:** Schedule status → CANCELLED, Harvest status → UNSCHEDULED
- **Errors:** ScheduleNotFoundError, ScheduleNotCancellableError
- **Events emitted:** ScheduleCancelledEvent (consumed by Crop domain to revert Harvest to UNSCHEDULED)

### GetSchedule
- **Trigger:** user action
- **Input:** scheduleId
- **Rules:** only return non-soft-deleted records. Include all operations and their inputs.
- **Success:** Schedule with operations and inputs, Harvest info (field, crop type, variety, dates)
- **Errors:** ScheduleNotFoundError
- **Events emitted:** none

### ListSchedules
- **Trigger:** user navigates to schedules page
- **Input:** pagination params, optional filters (status, fieldId, harvestId)
- **Rules:** only return non-soft-deleted records, paginated
- **Success:** paginated list of schedules with Harvest summary (field name, crop type, variety, dates, status)
- **Errors:** none
- **Events emitted:** none

---

### AddScheduleOperation
- **Trigger:** user action
- **Input:** scheduleId, date, type (SPRAYING | FERTIGATION)
- **Rules:** Schedule must exist and not be COMPLETED or CANCELLED. Date must be within Harvest period.
- **Success:** ScheduleOperation created
- **Errors:** ScheduleNotFoundError, ScheduleNotEditableError, OperationDateOutOfRangeError
- **Events emitted:** ScheduleOperationAddedEvent

### EditScheduleOperation
- **Trigger:** user action
- **Input:** operationId, date (optional), type (optional)
- **Rules:** same as add — date must be within Harvest period if changed. Schedule must not be COMPLETED or CANCELLED.
- **Success:** ScheduleOperation updated
- **Errors:** ScheduleOperationNotFoundError, ScheduleNotEditableError, OperationDateOutOfRangeError
- **Events emitted:** ScheduleOperationUpdatedEvent

### RemoveScheduleOperation
- **Trigger:** user action
- **Input:** operationId
- **Rules:** Schedule must not be COMPLETED or CANCELLED. Cascade deletes all ScheduleOperationInputs.
- **Success:** ScheduleOperation and its inputs removed
- **Errors:** ScheduleOperationNotFoundError, ScheduleNotEditableError
- **Events emitted:** ScheduleOperationRemovedEvent

---

### AddScheduleOperationInput
- **Trigger:** user action
- **Input:** operationId, inputId, dosagePer100L
- **Rules:** Operation must exist. Input must exist and be active. dosagePer100L > 0. Schedule must not be COMPLETED or CANCELLED.
- **Success:** ScheduleOperationInput created
- **Errors:** ScheduleOperationNotFoundError, InputNotFoundError, ScheduleNotEditableError
- **Events emitted:** ScheduleOperationInputAddedEvent

### EditScheduleOperationInput
- **Trigger:** user action
- **Input:** operationInputId, inputId (optional), dosagePer100L (optional)
- **Rules:** same validation as add. Schedule must not be COMPLETED or CANCELLED.
- **Success:** ScheduleOperationInput updated
- **Errors:** ScheduleOperationInputNotFoundError, InputNotFoundError, ScheduleNotEditableError
- **Events emitted:** ScheduleOperationInputUpdatedEvent

### RemoveScheduleOperationInput
- **Trigger:** user action
- **Input:** operationInputId
- **Rules:** Schedule must not be COMPLETED or CANCELLED.
- **Success:** ScheduleOperationInput removed
- **Errors:** ScheduleOperationInputNotFoundError, ScheduleNotEditableError
- **Events emitted:** ScheduleOperationInputRemovedEvent

---

### CopyScheduleOperationsToDay
- **Trigger:** user action (copy operations from one day to another)
- **Input:** scheduleId, sourceDate, targetDate
- **Rules:** Both dates must be within Harvest period. Schedule must not be COMPLETED or CANCELLED. Copies all operations and their inputs from sourceDate to targetDate. Does NOT remove existing operations on targetDate — adds to them.
- **Success:** New operations created on targetDate (copies of sourceDate operations with their inputs)
- **Errors:** ScheduleNotFoundError, ScheduleNotEditableError, OperationDateOutOfRangeError, NoOperationsOnSourceDateError
- **Events emitted:** ScheduleOperationAddedEvent (one per copied operation)

### CopySchedule
- **Trigger:** user action (copy entire schedule to create a new one for a different harvest)
- **Input:** sourceScheduleId, targetHarvestId
- **Rules:** Source Schedule must exist. Target Harvest must exist with status UNSCHEDULED. Target Harvest must not already have a Schedule. Copies all operations and inputs, adjusting dates relative to the new Harvest period. If source and target Harvest periods differ in length, only copy operations that fit within the target period.
- **Success:** New Schedule created with copied operations and inputs
- **Errors:** ScheduleNotFoundError, HarvestNotFoundError, HarvestNotUnscheduledError, ScheduleAlreadyExistsForHarvestError
- **Events emitted:** ScheduleCreatedEvent

## Cross-Domain Dependencies

- **Depends on:** Harvest (by `harvestId` — 1:1, inherits field/crop/dates), Input from Inventory (by `inputId` — references active inputs)
- **Emits events consumed by:** Crop (ScheduleActivatedEvent → activate Harvest, ScheduleCancelledEvent → revert Harvest to UNSCHEDULED), FieldTicket (schedule data feeds field ticket generation), Audit (all mutation events)

## Constraints

- One Schedule per Harvest (1:1) — enforced by unique constraint on harvestId
- Operation dates must be within Harvest period (startDate to expectedEndDate)
- Dosage always displayed in smaller unit (kg→g, L→mL, g→g, mL→mL, un→un)
- Dosage is per 100L of water
- All operations audited via domain events
- Hard delete cascades to operations, operation inputs, and draft FieldTickets
- Soft delete when printed/finalized FieldTickets exist
- Schedule status changes drive Harvest status changes (ACTIVE, CANCELLED→UNSCHEDULED)

## Edge Cases

- User creates Schedule for a Harvest that already has a cancelled Schedule → allowed (Harvest is UNSCHEDULED again)
- User adds operation on a date outside Harvest period → rejected with OperationDateOutOfRangeError
- User tries to activate Schedule with no operations → allowed (empty schedule can be activated — operations can be added later)
- User cancels active Schedule → Harvest reverts to UNSCHEDULED, new Schedule can be created
- User deletes Schedule with draft FieldTickets → hard delete cascades to FieldTickets
- User deletes Schedule with printed FieldTickets → soft delete (preserves history)
- User copies Schedule to a shorter Harvest period → only operations within the new period are copied, others are silently skipped
- User copies operations from a day to another day that already has operations → new operations are added alongside existing ones (no replacement)
- Concurrent Schedule creation for same Harvest → database unique constraint prevents duplicates
- Input becomes inactive after being assigned to an operation → existing assignment preserved, but cannot add new assignments for that input

## Harvest Domain Changes Required

When implementing Schedule, the following changes to the Crop domain (Harvest entity) are required:

1. **Remove `PLANNED` status** — replace with `UNSCHEDULED`
2. **New lifecycle:**
   ```
   UNSCHEDULED → ACTIVE (via ScheduleActivatedEvent, not manual)
   UNSCHEDULED → CANCELLED
   ACTIVE → COMPLETED (manual, but Schedule must be COMPLETED)
   ACTIVE → CANCELLED
   ```
3. **Remove manual `ActivateHarvest` use case** — activation is driven by Schedule domain
4. **Add event subscriber** for ScheduleActivatedEvent (→ activate Harvest) and ScheduleCancelledEvent (→ revert Harvest to UNSCHEDULED)
5. **`CompleteHarvest` rule change** — Schedule must be COMPLETED before Harvest can be completed
6. **Initial status on creation** — Harvest is created with status `UNSCHEDULED` (no schedule yet)
