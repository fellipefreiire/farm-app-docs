# User — Business Rules

## Responsibility
Manages user identity, authentication, and account lifecycle (creation, login, logout, soft-deletion, status toggle).

## Out of scope
- Audit logging (handled by `audit-log` domain via domain events)
- Authorization policies (handled by `infra/auth` with CASL)
- Password reset / email verification (not yet implemented)

## Entities

### User (Aggregate Root)
| Attribute | Type | Required | Rules |
|-----------|------|----------|-------|
| id | UUID | yes | generated by system |
| name | string | yes | — |
| email | string | yes | must be unique across all users |
| password | string | yes | stored as hash, never plaintext |
| role | enum (ADMIN, OWNER, MANAGER) | yes | defaults to OWNER on creation |
| active | boolean | yes | defaults to true on creation |
| createdAt | Date | yes | set on creation |
| updatedAt | Date | no | set on any mutation |
| deletedAt | Date | no | set on soft-delete |

**Invariants:**
- Email must be unique — enforced at use case level (check before create) and DB level (unique constraint)
- Password is always hashed before persistence — domain never stores plaintext
- A soft-deleted user (deletedAt is set) cannot be found by `findActiveById`

**Lifecycle:**
- Created (active: true, deletedAt: null) → Active
- Active → Inactive (via `toggleActive`)
- Inactive → Active (via `toggleActive`)
- Active/Inactive → Soft-deleted (via `softDelete`, sets deletedAt)

## Value Objects
None — email and password are stored as primitives. If validation rules grow, extract to value objects.

## Use Cases

### AuthenticateUser
- **Trigger:** User submits login form (POST /sessions)
- **Input:** email, password
- **Rules:** Email must match an existing user; password must match stored hash
- **Success:** `{ accessToken, refreshToken, expiresIn }`
- **Errors:** `WrongCredentialsError` (invalid email or password)
- **Events emitted:** `UserLoggedInEvent`

### CreateUser
- **Trigger:** Admin creates a new user (POST /users)
- **Input:** name, email, password, role (optional), actorId
- **Rules:** Email must not already exist
- **Success:** `{ data: User }`
- **Errors:** `UserAlreadyExistsError` (duplicate email)
- **Events emitted:** `UserCreatedEvent`

### LogoutUser
- **Trigger:** User logs out (POST /sessions/logout)
- **Input:** userId, actorId
- **Rules:** User must exist
- **Success:** `{ data: null }`
- **Errors:** `ResourceNotFoundError` (user not found)
- **Events emitted:** `UserLoggedOutEvent`

### RefreshUserToken
- **Trigger:** Client sends expired access token with valid refresh token (POST /sessions/refresh)
- **Input:** refreshToken
- **Rules:** Token must be valid, JTI must exist in token store
- **Success:** `{ accessToken, refreshToken, expiresIn }`
- **Errors:** `UserUnauthorizedError` (invalid or revoked token)
- **Events emitted:** none

## Cross-Domain Dependencies
- Depends on: none
- Emits events consumed by: `audit-log` (UserCreatedEvent, UserLoggedInEvent, UserLoggedOutEvent, UserDeletedEvent, UserStatusChangedEvent)

## Authorization
- `AuthenticateUser`: public (no auth required)
- `RefreshUserToken`: public (no auth required, token-based)
- `CreateUser`: ADMIN only (system admin creates users for tenants)
- `LogoutUser`: authenticated user (self)

## Constraints
- Unique email (DB unique constraint + use case check)
- Soft-delete (deletedAt field, no hard delete)
- All mutations dispatch domain events for audit trail

## Edge Cases
- Creating a user with an email that belongs to a soft-deleted user — currently blocked by uniqueness constraint
- Toggling status of a soft-deleted user — not prevented at domain level, consider adding guard
